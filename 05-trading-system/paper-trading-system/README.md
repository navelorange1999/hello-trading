# 5.1 模拟交易系统

## 为什么做这个项目？

这是整个学习路线的高潮——你要把前四章造的所有零件，组装成一个**能自动运行的交易系统**。

但注意：我们做的是**模拟交易**（paper trading），不涉及真金白银。这不是因为技术上做不到实盘，而是因为：
1. 你还没有足够的数据证明你的系统能赚钱
2. 模拟交易和实盘之间还有很多工程问题需要解决
3. 用模拟盘先跑一段时间，是负责任的做法

你的目标是「睡后收入」——这个系统就是你迈向那个目标的第一个真正的原型。

## 已知领域

- 数据管道（Chapter 1）
- 分析方法和策略（Chapter 2-3）
- 风险管理（Chapter 4）
- Python 模块化编程

## 新领域

- 端到端系统设计：数据 → 信号 → 执行 → 监控
- 定时任务和自动化
- 系统监控和异常告警
- 模拟执行与真实执行的差距

## Task 1: 设计端到端系统架构

**做什么**: 画出你的交易系统的完整架构图。定义每个模块的职责、接口和数据流向。

**思考提示**:
- 一个完整的交易系统至少有哪些模块？（数据、信号、风控、执行、监控、日志...）
- 这些模块之间应该如何通信？直接函数调用？消息队列？文件？每种方式的优劣是什么？
- 如果某个模块出错（比如数据源挂了），整个系统应该怎么反应？崩溃？跳过？告警？
- 你打算让系统跑在哪里？本地电脑？云服务器？这个选择影响架构设计吗？

**验证**: 你有一张清晰的架构图和一份模块说明文档。另一个有经验的开发者看了能理解你的设计。

📚 **关键资料**:
- 搜索 "algorithmic trading system architecture" — 看看成熟系统的架构参考
- 搜索 "modular design principles" — 松耦合、高内聚的设计原则
- 不需要过度设计——你的第一版系统应该是最简单的可用版本

---

## Task 2: 实现信号生成模块

**做什么**: 把你在 Chapter 3 中开发的策略封装为信号生成模块。它接收最新的市场数据，输出交易信号（买/卖/持有）。

**思考提示**:
- 信号模块应该只关心「发什么信号」，不关心「怎么执行」——这种分离为什么重要？
- 如果你想同时运行多个策略，信号模块应该怎么组织？策略间的信号冲突怎么处理？
- 信号应该只是「买/卖」，还是应该包含更多信息（如目标仓位比例、信心度）？
- 信号生成出错时（比如数据不足），应该输出什么？「不确定」也是一种有价值的信号。

**验证**: 模块能接收实时（或最新的）市场数据，产出明确的交易信号，并且信号逻辑与你回测中的策略完全一致。

---

## Task 3: 实现模拟执行和持仓管理

**做什么**: 接收交易信号后，模拟执行交易。维护一个虚拟账户，跟踪现金、持仓和总资产。

**思考提示**:
- 模拟执行和真实执行的最大区别是什么？（提示：滑点、部分成交、延迟）
- 你怎么模拟这些真实世界的摩擦？完全忽略？还是加入一些随机性？
- 持仓管理需要考虑什么？买入后持仓成本怎么计算？多次加仓后呢？
- 你的风险管理模块（Chapter 4）应该在执行之前还是之后检查？

**验证**: 虚拟账户的状态在每次交易后正确更新。一天结束后能生成持仓报告。

---

## Task 4: 实现监控与告警

**做什么**: 实现系统状态的监控和异常告警。当出现重要事件（大额交易、回撤超阈值、数据获取失败）时，发送通知。

**思考提示**:
- 什么事件值得告警？太少会错过问题，太多会让你对告警麻木（boy who cried wolf）。
- 通知渠道选什么？邮件、微信、Telegram？哪个最适合你的使用场景？
- 你在睡觉的时候系统在跑——你第二天早上打开手机，希望看到什么样的摘要信息？
- 如果系统完全失联了（比如服务器宕机），你怎么知道？

**验证**: 系统在触发预设条件时成功发送通知。你收到了测试告警消息。

📚 **关键资料**:
- 搜索 "python send email smtplib" — 最简单的告警方式
- 搜索 "python telegram bot tutorial" — Telegram Bot 是轻量级告警的好选择
- 搜索 "python schedule library" 或 "cron job" — 定时任务的实现方式

---

## Task 5: 运行一周模拟交易

**做什么**: 让你的系统真正运行一周。每天（或每几小时）自动获取数据、生成信号、模拟执行、发送报告。

**思考提示**:
- 一周的模拟交易够得出结论吗？不够——但你能发现系统中的工程问题。
- 在这一周中，系统出了哪些意料之外的问题？数据获取失败？策略信号异常？
- 你发现自己有没有忍不住想「手动干预」系统的冲动？这种冲动意味着什么？
- 模拟交易的结果和你的回测预期一致吗？如果不一致，差异来自哪里？

**验证**: 一周结束后，产出一份完整的模拟交易报告，包含：交易记录、资金变化、遇到的问题和你的反思。

---

## 完成标准

- **行为验证**: 系统能连续运行一周，自动完成数据获取→信号生成→模拟执行→报告发送的全流程
- **解释验证**: 你能列出模拟交易和真实交易之间至少 5 个重要差异
- **迁移验证**: 如果有人给你一个新策略，你能在 1 小时内将它接入你的系统

## 延伸思考

你现在有了一个能运行的模拟交易系统。从模拟到实盘还需要什么？

- **经纪商 API 接入**（如 Interactive Brokers、币安 API）
- **合规和税务考量**
- **更严格的错误处理和容错机制**
- **更长时间的模拟验证**（一周远远不够，至少应该跑 3-6 个月）

不要急于实盘。一个在模拟盘上运行了半年、经历了不同市场环境、你对它有充分信心的系统，才值得投入真金白银。
